# Android 双摄行车记录仪需求文档与开发计划

## 1. 项目概述

本项目拟开发一款 Android 行车记录仪应用，将手机固定在车内后，优先使用后置摄像头记录车辆前方道路，并在设备支持的情况下允许同时开启前置摄像头记录车内或驾驶员状态。应用需要支持循环录制、紧急锁定、后台持续录制、录制文件管理、事故取证导出等能力。

由于 Android 设备的摄像头硬件、厂商 HAL、系统版本和并发摄像头能力差异较大，前后摄像头“同时开启/同时录制”不能作为所有设备的默认能力。本项目必须先判断手机是否支持前后摄像头同时开启，再在设置中提供“同时开启前后摄像头”的开关。该开关仅在设备检测通过时可用，不支持的设备默认使用单后摄模式。

## 2. 产品目标

1. 将闲置 Android 手机作为低成本行车记录仪使用。
2. 在设备支持时同时记录车外道路画面和车内画面，提升事故、纠纷、驾驶行为分析的证据完整性。
3. 支持长时间稳定录制，自动分段、循环覆盖，减少用户手动维护成本。
4. 在设备不支持前后摄像头同时开启时，隐藏或禁用双摄开关，并提供稳定的单摄录制模式。

## 3. 目标用户

1. 私家车车主：希望用旧手机替代传统行车记录仪。
2. 网约车、出租车、货运司机：需要同时记录车外和车内情况。
3. 新手司机：希望保留驾驶过程影像以便复盘。
4. 车队管理者：需要后续扩展到批量设备和云端管理。

## 4. 核心使用场景

### 4.1 开车前快速启动

用户将手机固定在支架上，打开应用，应用自动检测前后摄像头是否支持同时开启，进入录制预览界面。用户点击录制，或连接车载蓝牙/充电器后自动开始录制。

### 4.2 双画面行车录制

当设备支持且用户在设置中打开“同时开启前后摄像头”开关后，应用同时录制后置摄像头和前置摄像头视频。界面默认以后置道路画面为主预览，前置画面以小窗显示。录制文件按时间分段保存。若设备不支持或用户关闭该开关，应用仅使用后置摄像头录制。

### 4.3 循环录制

当存储空间低于阈值时，应用自动删除最旧的普通录像文件。被用户锁定或被碰撞事件标记的视频不得自动删除。

### 4.4 紧急事件锁定

用户点击紧急按钮，或应用通过加速度传感器检测到疑似急刹、碰撞、剧烈震动后，将当前片段和前后相邻片段标记为锁定，避免循环覆盖。

### 4.5 后台持续录制

用户切到后台或锁屏后，应用应继续录制。系统通知栏显示录制状态、已录制时长、剩余空间，并提供停止录制入口。

### 4.6 录像回看与导出

用户可以按时间线查看录像，播放前后摄像头视频，导出单个片段、合并片段，或分享给保险、交警、平台客服。

## 5. 功能需求

### 5.1 设备能力检测与双摄开关状态

优先级：P0

需求：

1. 启动时检测设备是否支持前后摄像头同时开启。
2. 检测可用摄像头 ID、朝向、支持分辨率、帧率、编码能力。
3. 检测系统版本、摄像头权限、麦克风权限、通知权限、后台限制。
4. 将设备能力分为：
   - A 级：支持前后摄像头同时开启、同时预览和同时录制。
   - B 级：支持前后摄像头同时开启和同时预览，但录制能力受限。
   - C 级：仅支持单摄录制，可在前后摄像头间切换。
   - D 级：摄像头能力异常或权限不足，无法录制。
5. 当不支持前后摄像头同时开启时，明确提示原因，并禁用设置中的双摄开关。
6. 当支持前后摄像头同时开启时，设置中的双摄开关可用，用户可自行开启或关闭。
7. 双摄开关默认关闭，首次检测到设备支持双摄时，可通过弹窗或设置提示引导用户手动开启。
8. 设置页提供“重新检测双摄能力”入口，用于系统升级、权限变化或摄像头状态变化后的重新判断。
9. 录制过程中不允许切换双摄开关；用户修改时提示需停止当前录制后生效。
10. 双摄开关状态至少包含：检测中、可用未开启、可用已开启、不可用、检测失败。

验收标准：

1. 首次启动能在 3 秒内完成基础能力检测。
2. 不支持双摄的设备不会崩溃，不会进入黑屏录制状态。
3. 用户能在设置页查看当前设备支持的录制模式和双摄开关状态。
4. 不支持前后摄像头同时开启时，双摄开关展示为不可用，并显示原因说明。
5. 支持前后摄像头同时开启时，用户打开开关后，下一次开始录制进入双摄模式。
6. 用户点击重新检测后，设置页能更新双摄开关状态，并保留最近一次检测结果。
7. 录制中修改双摄开关不会影响当前录制链路。

### 5.2 双摄预览

优先级：P1

需求：

1. 仅当设备支持前后摄像头同时开启，且用户开启双摄开关时，启用后摄主画面 + 前摄小窗预览。
2. 支持切换主副画面。
3. 支持前摄镜像预览开关。
4. 支持横屏、竖屏和固定横屏驾驶模式。
5. 支持预览比例适配，避免画面拉伸。

验收标准：

1. 预览画面无明显变形。
2. 横竖屏切换后录制不中断，若设备不支持则应限制方向并说明。
3. 前后预览延迟满足驾驶场景可接受范围，目标小于 500ms。

### 5.3 双摄录制

优先级：P1

需求：

1. 仅当设备支持前后摄像头同时开启，且用户开启双摄开关时，支持前后摄像头同时录制独立视频文件。
2. 默认推荐参数：
   - 后摄：1080p / 30fps / H.264。
   - 前摄：720p 或 1080p / 30fps / H.264。
   - 音频：AAC，仅在用户开启“行车环境声录制”开关并授权麦克风权限后录制。
3. 支持录制参数按设备能力自动降级。
4. 支持固定分段时长，默认每 3 分钟一个片段，可配置 1、3、5 分钟。
5. 支持录制状态恢复，异常中断后保留已完成片段。
6. 文件命名包含时间、摄像头方向、锁定状态，例如 `20260703_143000_rear.mp4`。
7. 若用户关闭双摄开关，应用不得打开前置摄像头，也不得创建前摄录制文件。
8. 若双摄开关开启但录制启动失败，应用应自动关闭本次前摄录制并降级为后摄录制，同时提示用户。
9. 若用户关闭“行车环境声录制”开关，应用不得请求或占用麦克风，不得写入音频轨道。

验收标准：

1. 支持设备且双摄开关开启时，前后视频时间戳误差目标小于 300ms。
2. 支持设备且双摄开关开启时，连续录制 2 小时无崩溃、无明显音画不同步。
3. 任一摄像头录制失败时，应用提示并保留另一摄像头录制能力。
4. 双摄开关关闭时，录制流程只占用后置摄像头。
5. 行车环境声开关关闭时，生成的视频不包含音频轨道。

### 5.4 循环录制与存储管理

优先级：P0

需求：

1. 用户可自定义视频最大存储容量，例如 5GB、10GB、20GB，或输入自定义容量。
2. 低于保留空间阈值时自动删除最旧普通片段。
3. 锁定片段、收藏片段不得自动删除。
4. 支持按天、按摄像头方向、按锁定状态筛选文件。
5. 支持显示当前占用空间、预计可录制时长。
6. 自定义容量需设置最小值和最大值，建议最小 2GB，最大不超过应用可用存储空间的 80%。
7. 当用户设置的容量小于当前已占用容量时，应用提示将触发普通片段清理，并要求用户确认。
8. 双摄和环境声开启时，预计可录制时长应根据当前码率、音频开关和单双摄模式动态估算。

验收标准：

1. 存储接近上限时不会导致录制失败。
2. 循环删除仅影响普通片段。
3. 删除逻辑有事务保护，避免元数据和文件状态不一致。
4. 用户修改自定义视频存储容量后，设置立即保存，并在下一次清理策略中生效。
5. 当前占用空间超过新容量时，应用只删除普通片段，不删除锁定片段。

### 5.5 紧急锁定

优先级：P0

需求：

1. 录制界面提供明显的紧急锁定按钮。
2. 支持基于加速度传感器的疑似碰撞检测。
3. 事件触发后锁定当前片段、前一个片段和后一个片段。
4. 事件记录包含触发时间、传感器强度、GPS 位置。

验收标准：

1. 手动锁定响应时间小于 500ms。
2. 被锁定文件不会被循环删除。
3. 误触发可在历史记录中取消锁定。

### 5.6 后台录制与通知

优先级：P0

需求：

1. 使用前台服务保持录制。
2. 通知栏显示录制状态、时长、当前模式和停止按钮。
3. Android 13 及以上请求通知权限。
4. Android 14 及以上适配摄像头前台服务类型要求。
5. 处理省电策略、后台限制、锁屏后的摄像头占用问题。

验收标准：

1. 应用切后台后录制不中断。
2. 用户可从通知栏停止录制。
3. 权限不足时给出明确引导。

### 5.7 GPS 与驾驶信息叠加

优先级：P1

需求：

1. 记录 GPS 轨迹、速度、方向、时间。
2. 支持在回放界面展示速度和路线。
3. 可选将时间、水印、速度叠加到导出视频。
4. 支持关闭位置记录以保护隐私。

验收标准：

1. GPS 数据与视频片段可按时间关联。
2. 导出视频水印内容正确。
3. 用户关闭位置权限后核心录制仍可使用。

### 5.8 回放、导出与分享

优先级：P1

需求：

1. 支持视频列表、时间线、事件列表。
2. 支持后摄和前摄同步回放。
3. 支持导出：
   - 仅后摄视频。
   - 仅前摄视频。
   - 前后双文件打包。
   - 画中画合成视频。
4. 支持系统分享面板。

验收标准：

1. 10 秒内可打开最近一天视频列表。
2. 导出进度可见，可取消。
3. 导出失败时不破坏原文件。

### 5.9 设置

优先级：P0

需求：

1. 提供“同时开启前后摄像头”开关。
2. 双摄开关状态由设备能力检测结果控制：
   - 设备支持前后摄像头同时开启：开关可用，默认关闭。
   - 设备不支持前后摄像头同时开启：开关不可用，并展示“不支持当前设备”的原因。
   - 权限不足或检测失败：开关不可用，并引导用户授权或重新检测。
3. 录制模式：仅后摄、仅前摄、自动；双摄模式由“同时开启前后摄像头”开关控制。
4. 分辨率、帧率、码率配置。
5. 分段时长配置。
6. 自定义视频存储容量配置，支持预设值和手动输入。
7. 自动开始录制：打开应用、连接充电器、连接指定蓝牙设备。
8. 前摄镜像、行车环境声录制、水印、GPS 开关。
9. 碰撞检测灵敏度。
10. 保存用户最近一次双摄开关选择；设备能力变化、系统升级或摄像头异常时重新检测并更新开关可用状态。
11. 展示最近一次双摄能力检测时间、检测结果和失败原因。
12. 行车环境声录制开关默认关闭；用户首次打开时请求麦克风权限，并说明仅用于本地行车视频录音。
13. 用户关闭行车环境声录制后，后续录制不包含音频轨道。

验收标准：

1. 双摄开关变更在下一次开始录制时生效；录制中修改时提示需停止后重新开始。
2. 设备不支持前后摄像头同时开启时，用户无法强行打开双摄开关。
3. 设置变更在下一段录制生效，必要时提示需重启录制。
4. 非法参数组合不可选。
5. 恢复默认设置可用，恢复后双摄开关回到默认关闭状态。
6. 重新检测期间双摄开关进入检测中状态，检测完成后自动刷新可用性。
7. 自定义视频存储容量不能低于最小容量，也不能超过当前设备可用上限。
8. 行车环境声录制开关关闭时，应用不请求麦克风权限；打开时若用户拒绝权限，开关回到关闭状态并提示原因。

## 6. 非功能需求

### 6.1 稳定性

1. 连续录制目标：至少 4 小时稳定运行。
2. 摄像头占用失败、编码器失败、存储不足时应可恢复或降级。
3. 应用崩溃后，下次启动能扫描并修复未完成片段元数据。

### 6.2 性能

1. 双摄开关开启时 CPU 使用率应尽量低于 60%，避免过热。
2. 优先使用硬件编码。
3. 支持根据温度、电量、性能状态自动降级分辨率或停止前摄录制。

### 6.3 功耗与发热

1. 充电时长时间录制需监控温度。
2. 设备过热时提示用户，并按策略降级；双摄场景优先关闭前摄录制。
3. 支持熄屏录制，但要验证不同厂商系统兼容性。

### 6.4 隐私与安全

1. 视频默认仅保存在本机。
2. 不默认上传云端。
3. 首次使用时说明前摄、麦克风、位置数据用途。
4. 导出和分享由用户主动触发。
5. 行车环境声录制默认关闭，只有用户主动开启并授权后才采集麦克风。

### 6.5 兼容性

1. 最低支持版本建议 Android 10。
2. 重点适配 Android 12、13、14、15、16。
3. 厂商测试覆盖：Pixel、Samsung、小米、OPPO、vivo、荣耀、一加。

## 7. 技术方案

### 7.1 推荐架构

1. UI 层：Jetpack Compose。
2. 摄像头层：
   - 首选 CameraX，降低常规预览和录制复杂度。
   - 对双摄并发能力要求高的部分，保留 Camera2 适配通道。
3. 录制层：MediaCodec / MediaMuxer 或 CameraX VideoCapture，按设备能力选择。
4. 数据层：Room 保存片段、事件、锁定状态、GPS 元数据。
5. 存储层：MediaStore + 应用私有索引。
6. 后台任务：Foreground Service。
7. 传感器层：SensorManager 采集加速度。
8. 位置层：Fused Location Provider 或系统 LocationManager。

### 7.2 双摄实现策略

第一阶段以能力探测和设置开关为核心：

1. 使用 Camera2 `CameraManager` 获取摄像头列表和并发组合能力。
2. 若设备报告支持前后摄像头并发，则尝试建立前后双 camera session。
3. 使用低规格组合进行探测，例如后摄 1080p + 前摄 720p。
4. 探测成功后，将“同时开启前后摄像头”开关标记为可用，但默认保持关闭。
5. 探测失败或设备不支持时，将双摄开关标记为不可用，并记录失败原因。
6. 将探测结果持久化，用于下次启动快速展示；进入设置页或权限变化后允许重新检测。
7. 双摄开关只控制下一次录制启动时是否初始化前置摄像头，不在录制中动态切换摄像头链路。

第二阶段实现受开关控制的录制：

1. 双摄开关关闭时，只初始化后置摄像头录制链路。
2. 双摄开关开启且设备支持时，前后摄像头分别输出到独立编码器。
3. 每个摄像头独立分段，使用统一片段组 ID 关联同一时间窗口内的前后视频。
4. 用户开启行车环境声录制后，音频只录制一路，默认绑定到后摄片段组；用户关闭时不创建音频输入和音频轨道。
5. 回放时按片段组同步播放。

第三阶段提供降级：

1. 双摄开关开启但双摄启动失败：本次降级为后摄录制，并提示用户。
2. 前摄失败：后摄继续录制，并建议用户关闭双摄开关。
3. 过热或编码压力过高：降低前摄分辨率或停止前摄。
4. 设备完全不支持并发：禁用双摄开关，使用单摄模式。

### 7.3 文件与元数据设计

建议目录：

```text
Dashcam/
  normal/
    2026-07-03/
      group_143000/
        20260703_143000_rear.mp4
        20260703_143000_front.mp4
        metadata.json
  locked/
    2026-07-03/
```

核心数据表：

1. `recording_group`：片段组、开始时间、结束时间、模式、锁定状态。
2. `video_segment`：文件路径、摄像头方向、分辨率、码率、时长、大小。
3. `event_marker`：紧急事件、触发类型、时间、传感器值、位置。
4. `gps_point`：时间、经纬度、速度、方向、精度。

## 8. 权限需求

1. `CAMERA`：摄像头录制。
2. `RECORD_AUDIO`：可选，仅在用户开启行车环境声录制时请求。
3. `POST_NOTIFICATIONS`：Android 13 及以上通知栏状态。
4. `ACCESS_FINE_LOCATION` / `ACCESS_COARSE_LOCATION`：可选，记录轨迹和速度。
5. 前台服务相关权限：用于后台录制和摄像头前台服务类型。
6. 存储访问：优先使用 MediaStore，避免申请宽泛文件权限。

## 11. 主要风险与应对

| 风险 | 等级 | 说明 | 应对 |
| --- | --- | --- | --- |
| 部分手机不支持前后摄像头同时开启 | 高 | Android 并不保证所有设备支持双摄同时开启或同时录制 | 启动能力检测，不支持时禁用双摄开关，提供单摄降级和兼容列表 |
| 长时间录制发热严重 | 高 | 双摄 + 编码 + 充电容易过热 | 自动降级分辨率，温度告警，停止前摄 |
| 后台录制被系统限制 | 高 | 厂商系统可能杀后台或限制摄像头 | 前台服务、通知、厂商引导、白名单说明 |
| 音视频同步困难 | 中 | 双编码器时间戳可能漂移 | 统一时钟源，片段组同步，回放校正 |
| 存储文件损坏 | 中 | 异常断电或崩溃会导致片段未完成 | 分段录制、临时文件、启动修复扫描 |
| 隐私争议 | 中 | 前摄和环境声涉及车内隐私 | 默认本地保存，明确权限说明，环境声录制默认关闭且可随时关闭 |

## 12. MVP 范围建议

MVP 必须包含：

1. 设备能力检测，判断是否支持前后摄像头同时开启。
2. 后摄录制。
3. 设置页“同时开启前后摄像头”开关。
4. 设备支持时开关可用，设备不支持时开关禁用。
5. 支持设备且用户开启开关后的前后双摄录制。
6. 单摄降级模式。
7. 分段保存。
8. 前台服务后台录制。
9. 基础录像列表。
10. 手动紧急锁定。
11. 设置页重新检测双摄能力。
12. 双摄启动失败后的自动回退和失败原因提示。
13. 自定义视频存储容量配置。
14. 行车环境声录制开关。

MVP 暂不包含：

1. 云端上传。
2. AI 驾驶行为分析。
3. 车队管理后台。
4. 自动车牌识别。
5. 复杂视频剪辑。

## 13. 验收指标

1. 应用能在首次启动或设置页重新检测时判断手机是否支持前后摄像头同时开启。
2. 在至少 3 台支持双摄并发的测试设备上，双摄开关可用，用户开启后可同时录制前后视频。
3. 在至少 3 台不支持双摄并发的设备上，双摄开关不可用，并可稳定使用单后摄录制。
4. 双摄开关关闭时，应用只占用后置摄像头。
5. 支持设备且双摄开关开启时，双摄连续录制 2 小时无崩溃。
6. 录制中切换双摄开关不会改变当前录制状态，并提示下次录制生效。
7. 双摄启动失败时，本次录制自动回退为后摄录制，并展示失败原因。
8. 循环录制不会删除锁定视频。
9. 后台录制期间通知常驻，用户可随时停止。
10. 用户可设置自定义视频存储容量，循环清理按该容量执行。
11. 行车环境声录制开关关闭时视频不包含音频轨道，开启并授权后视频包含环境声。
12. 录制文件可在系统相册或应用内正常播放。
